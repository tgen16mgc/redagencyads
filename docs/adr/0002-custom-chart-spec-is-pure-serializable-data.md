# Custom Chart spec is pure serializable data, and deserialize returns only render-ready specs

A Custom Chart is stored as a plain, JSON-serializable spec: an `id`, a `title`, a `type`, a fixed `xKey`, a `dualAxis` flag, and an array of Chart Series where each series is pure data (`key` + `axis`). No field holds a function or any non-serializable value. How a series is drawn — which Recharts element, which color, which tooltip formatter — is derived at render time from the spec, never stored on it.

This choice exists because Custom Charts persist to `localStorage` and must survive `JSON.stringify` → `JSON.parse` intact. A `render` function on a series (the obvious way to let a series decide how it draws) silently vanishes on the first reload, leaving a spec that looks complete in memory but is broken once restored. Keeping the spec pure data makes persistence, equality, validation, and testing trivial, at the cost of pushing per-series render decisions into the render layer. When per-series mixed rendering inside a composed chart is needed, it will be added as a serializable string enum (`seriesType: "line" | "bar"`), never as a function.

`deserializeCharts(raw)` is strict by construction: it returns only specs that are both structurally well-formed and currently render-legible (they pass Spec Validation). A stored entry is dropped in full — not trimmed — if any series references a metric outside the current Metric Catalog, if structure is malformed, or if the spec would not render legibly. Unparseable or non-array input yields an empty list. This means a renamed or removed metric discards the whole chart that used it rather than producing a silently altered chart the user never designed.

The trade-off is forgiveness versus trust. Trimming unknown series or keeping structurally-valid-but-illegible specs would preserve more of a user's saved work across schema changes, but it would fabricate charts the author never made and force every downstream consumer (renderer, list, print export) to re-validate defensively. We chose trust: every spec returned by `deserializeCharts` is render-ready, so downstream code consumes it blindly. The cost is that a metric rename silently drops affected saved charts and the user re-creates them — cheap, since charts are quick to rebuild and the alternative is a mangled chart that misrepresents data. If a future metric refactor is expected to invalidate many saved charts at once, a one-time migration that rewrites stored specs is the escape hatch, not loosening the deserialize gate.
